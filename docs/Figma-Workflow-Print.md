# Краткая выжимка: Figma Workflow и лимит запросов

## Цель

Краткий документ для передачи коллеге — **почему срабатывает лимит Figma**, **как мы запрашиваем данные**, и **как использовать кэш**.

---

## 1. Почему Figma лимитирует запросы (429)

**Причина:** Figma API (через MCP: Framelink, Figma Remote/Desktop) ограничивает количество запросов в единицу времени. Особенно строго лимитируется эндпоинт `/nodes` с параметром `nodeId` — каждый запрос по конкретному узлу считается отдельно.

**Что происходит:**

- AI делает много запросов подряд (`get_figma_data`, `get_design_context` для разных nodeId)
- Каждый компонент = 1–2+ запроса
- Лимит быстро исчерпывается → ответ **429 (Too Many Requests)**

---

## 2. Наш workflow: сначала вся страница → кэш → работа из кэша

**Двухэтапный подход:**

### Этап 1: Загрузка всего файла (минимальное число запросов)

- **Один раз** запрашиваем данные **без** параметра `nodeId` (или запрашиваем всю страницу целиком)
- Сохраняем результат в:
  - `nextjs/figma-data/full-figma-data-YYYY-MM-DD.yaml` (полный дамп)
  - `nextjs/figma-data/components/[имя].json` (выжимки по компонентам)

### Этап 2: Работа из локального кэша

- Перед любым запросом к Figma — **сначала проверяем** `FIGMA_CACHE.md` и `figma-data/`
- Если данные есть → **не запрашиваем Figma**, используем кэш
- Если компонент новый → запрашиваем → сохраняем в кэш

**Workaround при 429:** использовать `full-figma-data-*.yaml` — там уже есть всё дерево узлов, из него можно извлекать нужные компоненты без повторных запросов.

---

## 3. Правила работы (из skills)

1. **Перед запросом Figma** — проверить `nextjs/figma-data/FIGMA_CACHE.md`
2. **Если компонент в кэше** — не вызывать MCP, брать данные из кэша
3. **Иконки/изображения** — скачивать одним batch (`download_figma_images`) за раз, не по одной
4. **Layout-числа** (dimensions, gap, padding) — брать из raw Framelink (`get_figma_data` по nodeId блока); summarized JSON часто не содержит полных значений
5. **При 429** — не retry; использовать кэш, подождать 1–2 минуты

---

## 4. Структура кэша

| Путь                                | Назначение                                                |
| ----------------------------------- | --------------------------------------------------------- |
| `FIGMA_CACHE.md`                    | Реестр компонентов, структур, типографики, цветов, nodeId |
| `figma-data/components/*.json`      | Выжимки по компонентам                                    |
| `figma-data/full-figma-data-*.yaml` | Полный дамп файла Figma (обход 429)                       |
| `public/images/figma/`              | Скачанные SVG/PNG                                         |

---

## 5. Порядок вызовов (pixel-perfect)

1. **Framelink** `get_figma_data(fileKey, nodeId)` — layout-числа
2. **Figma** `get_design_context` — черновик структуры
3. **Framelink** `download_figma_images` — batch загрузка иконок
4. **Figma** `get_screenshot` — визуальная сверка

**Важно:** для каждого источника сначала проверять кэш; вызывать MCP только при отсутствии данных.

---

## Итог для коллеги

**Почему лимит:** много запросов по `nodeId` → 429.

**Что делать:** сначала проверить `FIGMA_CACHE.md` и `figma-data/`; если есть — не запрашивать. Для новых компонентов — запросить целиком файл или блок, сохранить в кэш, дальше работать из кэша. Иконки — batch, не по одной.
