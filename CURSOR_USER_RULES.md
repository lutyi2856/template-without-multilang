# User Rules для Cursor AI Assistant

> **Важно:** Эти правила нужно добавить в Cursor Settings → Rules → User Rules

Скопируйте содержимое этого файла и вставьте в User Rules настройки Cursor.

---

## Правило #1: Проверка работы перед отчетом

Перед тем как сообщить "Готово" или "Все работает":

### 1. Запустить приложение (если веб-проект)

- Проверить, что сервер запустился без ошибок
- Открыть в браузере (если применимо)

### 2. Сделать скриншот результата

- Использовать `browser_take_screenshot` для веб-страниц
- Показать визуальный результат пользователю
- Скриншот ОБЯЗАТЕЛЕН для UI изменений

### 2.5. Выполнить тесты и рекомендации (если применимо)

- Запустить Playwright: `npx playwright test`
- Реализовать все пункты из плана (outputFileTracingRoot, fetch cache и т.д.)
- «Готово» — только когда всё из плана сделано

### 3. Проверить функциональность

- Кликнуть на интерактивные элементы
- Проверить, что изменения применились
- Убедиться, что нет ошибок в консоли
- Проверить responsive design (mobile/tablet/desktop)

### 4. Проверить linter errors

- Запустить `read_lints` для измененных файлов
- Исправить ВСЕ новые ошибки
- НЕ оставлять warnings без объяснения

### 5. Только после проверки сообщать о выполнении

**Примеры:**

❌ **Плохо:**

```
Я изменил компонент. Готово!
```

✅ **Хорошо:**

```
Изменил компонент. Запускаю приложение...
[скриншот браузера]
Проверил:
- ✅ Компонент отображается правильно
- ✅ Интерактивность работает
- ✅ Нет ошибок в консоли
- ✅ Linter чист
Готово!
```

## Правило #2: Использование браузера для проверки UI

Когда работаю с UI компонентами:

1. `browser_navigate` - открыть страницу
2. `browser_snapshot` - проверить элементы
3. `browser_click` / `browser_type` - проверить интерактивность
4. `browser_take_screenshot` - визуальная проверка
5. Показать скриншот пользователю

**Пример workflow:**

```
1. Изменил компонент Button
2. browser_navigate("http://localhost:3000")
3. browser_snapshot() - проверить элементы
4. browser_click на кнопку
5. browser_take_screenshot()
6. Показать скриншот пользователю
7. Отчитаться о выполнении
```

## Правило #3: Минимизация запросов к внешним API

Перед запросом к Figma MCP, WordPress MCP, или другим API:

1. Проверить локальный кэш
2. Проверить, что данные еще не были получены
3. Запросить только если данных нет
4. Сохранить результат в кэш

**Пример для Figma:**

```
1. Открыть nextjs/figma-data/FIGMA_CACHE.md
2. Найти компонент по Node ID
3. Если есть → использовать кэш
4. Если нет → запросить MCP → сохранить в кэш
```

**Почему это важно:**

- Экономия API calls
- Быстрее работа (нет ожидания)
- Меньше costs для пользователя

## Правило #4: Использование Skills

После решения сложной задачи или обсуждения паттернов:

1. Предложить пользователю: "Хотите сгенерировать правила из этого чата?"
2. Если да → использовать `/Generate Cursor Rules`
3. Проверить сгенерированные правила
4. Сохранить в `.cursor/rules/`

**Когда предлагать:**

- После решения сложной архитектурной задачи
- После обсуждения best practices
- После выработки новых паттернов
- Когда появились инсайты, которые стоит сохранить

**Пример:**

```
AI: Мы обсудили структуру компонентов. Хотите сгенерировать правила
    из этого чата? Введите /Generate Cursor Rules
```

## Правило #5: Контекст и производительность

Управление контекстом:

### Используй эти команды регулярно:

- `/Reset Context` - когда контекст перегружен (>50 файлов)
- `/Add Active Files to Context` - вместо всех файлов
- `@filename` - для точечного добавления файлов

### НЕ добавляй в контекст:

- ❌ Файлы из `node_modules/`
- ❌ Файлы из `.next/`, `dist/`, `build/`
- ❌ Файлы из `.git/`
- ❌ Large media files
- ❌ Все открытые файлы, если их >20

### Best Practice:

```
1. Перед началом задачи: /Reset Context
2. Открыть только нужные файлы
3. Добавить в контекст: /Add Active Files to Context
4. Работать
5. После завершения: /Reset Context
```

## Правило #6: Качество кода

Перед завершением задачи:

### Проверить:

1. ✅ TypeScript errors - нет ошибок компиляции
2. ✅ ESLint warnings - исправлены или объяснены
3. ✅ Prettier форматирование - код отформатирован
4. ✅ Project rules - код следует правилам проекта
5. ✅ Дублирование кода - нет copy-paste

### Как проверять:

```
1. read_lints для измененных файлов
2. Проверить вывод TypeScript
3. Убедиться, что Prettier применен
4. Сравнить с правилами в .cursor/rules/
```

### Если есть ошибки:

- Исправить перед отчетом
- Или объяснить, почему оставлены
- НЕ игнорировать ошибки

## Правило #7: Документация

При создании нового компонента/функции:

### Добавить:

1. ✅ JSDoc комментарий с описанием
2. ✅ Примеры использования (@example)
3. ✅ Описание параметров (@param)
4. ✅ Описание возвращаемого значения (@returns)

### НЕ создавать отдельные MD файлы без запроса

**Пример:**

```typescript
/**
 * Button - переиспользуемый компонент кнопки
 *
 * @example
 * <Button variant="primary" onClick={handleClick}>
 *   Click me
 * </Button>
 *
 * @param variant - Стиль кнопки: 'primary' | 'secondary'
 * @param onClick - Обработчик клика
 * @param children - Содержимое кнопки
 */
export function Button({ variant, onClick, children }: ButtonProps) {
  // ...
}
```

### Когда обновлять README:

- Добавлен новый значимый компонент
- Изменена архитектура
- Добавлена новая фича
- Пользователь явно просит

## Правило #8: Git и версионирование

### НЕ делать:

- ❌ Commit без явного запроса пользователя
- ❌ Использовать `--no-verify` или `--force`
- ❌ Изменять git config
- ❌ Push --force к main/master
- ❌ Skip hooks без разрешения

### Делать:

- ✅ Предлагать осмысленные commit messages
- ✅ Группировать связанные изменения в один commit
- ✅ Спрашивать перед commit, если не попросили явно
- ✅ Использовать conventional commits формат

**Примеры commit messages:**

```
✅ Хорошо:
feat: add doctor card component with video support
fix: correct spacing in rating badge
refactor: extract font configuration to design system

❌ Плохо:
update files
fix bug
changes
```

### Когда спрашивать:

```
AI: Я сделал следующие изменения:
    - Создал компонент DoctorCard
    - Обновил типы в types.ts
    - Добавил mock data

    Хотите сделать commit? Предлагаю commit message:
    "feat: add doctor card component with mock data"
```

## Как применить эти правила

### Шаг 1: Открыть Cursor Settings

1. Cmd/Ctrl + ,
2. Найти "Rules" в левом меню
3. Выбрать "User Rules"

1. Открыть этот файл (CURSOR_USER_RULES.md)
2. Скопировать все содержимое (начиная с "# User Rules...")
3. Вставить в поле "User Rules"

### Шаг 3: Сохранить

1. Нажать "Save"
2. Перезапустить Cursor (рекомендуется)

### Шаг 4: Проверка

1. Открыть новый чат
2. Попросить AI что-то сделать
3. Убедиться, что AI следует правилам

## Troubleshooting

### Правила не применяются

**Решение:**

1. Перезапустить Cursor
2. Проверить, что правила в User Rules, а не Project Rules
3. Убедиться, что нет синтаксических ошибок

### AI игнорирует правила

**Решение:**

1. Напомнить AI в чате: "Пожалуйста, следуй User Rules"
2. Проверить, что правила четкие и конкретные
3. Перезапустить Cursor

### Слишком много правил

**Решение:**

1. Оставить только самые важные (3-5 правил)
2. Объединить похожие правила
3. Удалить устаревшие
